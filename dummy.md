# 🚀 ささっとノート (Sasatto Note) 技術解説ドキュメント

ささっとノートは、完全にサーバーレス（静的サイト）でありながら、書いた記事データをURLだけで共有できる画期的なMarkdownエディタです。
本稿では、このアプリを支えるコア技術やデータ圧縮の仕組み、最適化のプロセスについて徹底的に解説します。

---

## 📌 プロジェクト概要とコンセプト

多くのMarkdownエディタは、記事データをデータベース（MySQLやMongoDB等）に保存するため、WebサーバーやAPIサーバーなどのインフラが必要になります。しかし、ささっとノートは以下のコンセプトを掲げて開発されました。

1. **インフラコストゼロ**: 静的Webホスティング（GitHub Pagesなど）だけで動作する。
2. **会員登録・ログイン不要**: ユーザーがアクセスした瞬間に執筆を開始できる。
3. **データ永続性の担保**: サーバーがない代わりに、ブラウザの `localStorage` への自動一時保存と、URLのフラグメント（`#q=...`）を使った永続共有を実現する。

---

## 🛠️ 技術スタック

ささっとノートは、モダンで軽量なフロントエンド技術によって構築されています。

*   **Vite + React + TypeScript**: 高速な開発フィードバックと堅牢な型安全性を確保。
*   **Vanilla CSS**: TailwindCSSなどのライブラリに依存せず、WordPressの投稿画面を模した美しいUIと、ダークモード、ツールチップ吹き出しを独自にスタイリング。
*   **marked**: Markdownテキストを高速かつ正確にHTMLへレンダリング。
*   **DOMPurify**: ユーザーが記述したMarkdown経由でのXSS（クロスサイトスクリプティング）脆弱性を防ぐためのHTMLサニタイザー。
*   **brotli-wasm**: ブラウザ上で高速にBrotli圧縮・解凍を行うためのWebAssemblyモジュール。
*   **canvas-confetti**: エクスポート成功時にUXを豊かにする紙吹雪アニメーションライブラリ。

## ⚡ URL共有を実現する「Brotli × Base64URL」

ささっとノートの最も特徴的な機能は、**「URLの長さ制限の中でどれだけ長い記事データを詰め込めるか」**という挑戦です。

### 1. データ共有の流れ
記事をエクスポートする際、裏側では以下の処理がミリ秒単位で実行されます。

```
[記事データ (JSON)] ─► [バイナリ圧縮] ─► [Base64URL化] ─► [URLフラグメントに結合]
```

1.  タイトルと本文を軽量なJSONオブジェクトに変換：`{"t":"タイトル","c":"本文"}`
2.  文字列データをUTF-8バイト配列（`Uint8Array`）に変換。
3.  **Brotliアルゴリズム**を用いてバイナリ圧縮。
4.  バイナリデータをURLにそのまま載せられる安全な半角英数字（`A-Z`, `a-z`, `0-9`, `-`, `_`）に変換する **Base64URLエンコード** を適用。
5.  URLの末尾（`#q=...`）にセット。

### 2. なぜ「文字数」と「URLの長さ」にギャップが生まれるのか？
日本語は1文字あたり通常 **3バイト** のデータ量（UTF-8）を持ちます。
一方、URLで使えるBase64URLの英数字は、1文字あたり **0.75バイト（6ビット）** 分の情報しか表現できません。

そのため、生データをそのままURL化（N方式）すると、元の日本語文字数の **約4倍** の長さのURLになってしまいます。
Brotliで1/3〜1/5に圧縮することで、Base64化による1.33倍の膨張を加味しても、元の日本語文字数と同等か、それ以下の長さのURLに抑え込むことができます。

## 🧠 最適化：バイナリ可変長辞書エンコーディング

Brotli単体でも非常に強力ですが、ささっとノートではさらに圧縮率を突き詰めるため、**「辞書置換（Tokenization）」** と **「Brotli」** を組み合わせたハイブリッド圧縮（H方式）を開発しました。

### 1. UTF-8の不正バイトを利用したエスケープ設計
テキストを別の文字列（例：`{123}`など）に置換すると、括弧や数字の分だけデータが膨らんでしまい、逆転現象が起きます。
そこで、ささっとノートでは、**UTF-8の規格上絶対に発生しない未使用バイト（0xF4 〜 0xFF）** をトークンマーカーとして利用し、1〜2バイトのバイナリデータへ直接置換するアルゴリズムを実装しました。

*   **最頻出の11語 (インデックス0〜10)**:
    `0xF5` 〜 `0xFF` の **1バイトのみ** に置き換え（サイズ：1バイト）。
*   **それ以外の拡張語彙（最大255語）**:
    `0xF4` ＋ `辞書ID` の **2バイト** に置き換え（サイズ：2バイト）。

### 2. 辞書トークン化（ハイブリッド方式）の効果
これにより、「ささっとノート」や「ダウンロード」、Markdown記法の「`### `」などの頻出語句が、わずか1〜2バイトに収まります。

```
[元の文字列] : "ささっとノート" (21バイト)
      │
      ▼ (バイナリエンコード)
[トークンデータ] : 0xF5 (1バイト) ── 21倍 of 圧縮！
```

このエンコードを行ったバイナリ列をBrotliに渡すことで、Brotliはより効率的にデータを圧縮でき、URLの文字数を限界まで削減することに成功しました。

### 3. C方式とH方式の自動選択
文章が短すぎる場合や、辞書に全くヒットしない文章の場合、辞書置換を行うことで逆に文脈が壊れて圧縮率が落ちることがあります。
そのため、アプリは内部で **C方式（Brotli単体）** と **H方式（ハイブリッド）** の両方を計算し、**「1文字でも短い方のURL」** をリアルタイムに自動選択して出力します。

## 🔍 圧縮パフォーマンス比較（実測データ）

夏目漱石の「私の個人主義」のダミーテキスト（4,067文字）を用いた実際のベンチマーク結果です。

| 方式 | URLの文字数 | 特徴 |
| :--- | :---: | :--- |
| **生データ (N)** | 16,217 文字 | 圧縮なし。URLが長すぎて共有不可。 |
| **Brotliのみ (C)** | 6,325 文字 | 標準的な圧縮。Brotliの内蔵辞書が機能。 |
| **ハイブリッド (H)** | **6,216 文字** 🏆 | バイナリ辞書 ＋ Brotli。最も短くなる最適値。 |

## 🔮 今後の展望と課題

ささっとノートは現在非常に安定して動作していますが、さらなる限界突破のために以下のロードマップを検討しています。

1.  **辞書の動的学習**:
    ユーザーがよく使う言葉をブラウザ側で学習し、ユーザーごとのカスタム辞書をURLのヘッダーに数バイトで格納する仕組み。
2.  **SentencePieceの超軽量サブワード辞書の導入**:
    初期ロードを損なわない範囲（10KB以下）で、統計的に最も圧縮率が高くなる語彙リストを算出し、完全にトークナイズされた整数配列としてBrotliに引きވަす仕組み。

ささっとノートは、サーバーレスという制約の中でデータの表現限界に挑戦し続ける、実験的かつ実用的なエディタです。
ぜひ、あなたの手で様々な文章を書いて、その圧縮力を体験してみてください！
